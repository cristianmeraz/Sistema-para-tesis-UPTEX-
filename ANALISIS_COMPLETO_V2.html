4. 🔄 FLUJO DE LA APLICACIÓN

Ciclo de Vida de un Request:

1. Cliente hace HTTP Request ↓ 2. Laravel recepciona en app.php ↓ 3. Verifica ruta: ├─ /api/... → Rutas API │ ├─ auth:sanctum │ ├─ role:X (si aplica) │ └─ Controller API │ └─ /... → Rutas Web ├─ web.auth ├─ web.admin/tecnico (si aplica) └─ Controller Web → View Blade ↓ 4. Controller procesa: ├─ Validación de datos ├─ Interacción con Modelos └─ Retorna JSON o HTML ↓ 5. Response al cliente

Ejemplo: Crear Ticket (Usuario Normal)

1. GET /tickets/create
   → WebController->create()
   → Retorna formulario Blade

2. POST /tickets
   → Validación
   → WebController->store()
   → Inserta en BD
   → Redirect a /tickets/{id}

3. GET /tickets/{id}
   → Muestra detalle con estado

2. 🚀 EAGER LOADING

Problema: Consultas N+1 en tickets

Solución:

// ANTES (MAL - N+1):
$tickets = Ticket::all();
foreach ($tickets as $ticket) {
    echo $ticket->usuario->nombre; // Query adicional por cada ticket
}

// DESPUÉS (BIEN - Eager Loading):
$tickets = Ticket::with('usuario', 'tecnicoAsignado', 'area', 'prioridad', 'estado')->get();
foreach ($tickets as $ticket) {
    echo $ticket->usuario->nombre; // Sin queries adicionales

14. 🔐 MEJORAS FASE 4 (SEGURIDAD AVANZADA)

1. ⏰ RATE LIMITING EN LOGIN

Requisito: Máximo 3 intentos fallidos = bloqueo temporal

Duración bloqueo: 15 minutos

// En AuthController->login()
$usuario = Usuario::where('email', $request->email)->first();

if (!$usuario || !Hash::check($request->password, $usuario->password)) {
    $usuario?->increment('login_intentos');
    
    if ($usuario && $usuario->login_intentos >= 3) {
        $usuario->update([
            'bloqueado_hasta' => now()->addMinutes(15)
        ]);
        return response()->json([
            'message' => 'Cuenta bloqueada por 15 minutos'
        ], 429);
    }
    throw ValidationException::withMessages([
        'email' => 'Credenciales inválidas'
    ]);
}

// Reset al login exitoso
$usuario->update([
    'login_intentos' => 0,
    'bloqueado_hasta' => null
]);

� GIT & GITHUB WORKFLOW (3 Colaboradores):

📌 Estrategia de Branching:

  • main: Código de producción (deployable)
  • develop: Rama integración (siempre estable)
  • feature/usuario-*: Ramas del Colaborador 1
  • feature/tecnico-*: Ramas del Colaborador 2
  • feature/admin-*: Ramas del Colaborador 3

💾 Commits por tarea:

  • ✅ Formato: git commit -m "feat(usuario): quitar prioridad y asistente IA"
  • ✅ Prefijos: feat, fix, refactor, chore, docs, style
  • ✅ Frecuencia: 1 commit por tarea completada
  • ✅ Mensajes descriptivos: Qué cambió y por qué
git add .
git commit -m "feat(usuario): remover selector de prioridad en formulario"
git push origin feature/usuario-limpieza

🔄 Flujo de Pull Requests (PR):

  1. Crear rama: git checkout -b feature/usuario-fase1
  2. Hacer commits: git commit -m "..."
  3. Push a GitHub: git push origin feature/usuario-fase1
  4. Abrir PR en GitHub (feature/usuario-* → develop)
  5. ✅ Descripción del PR:
    • Qué cambios hace
    • Fase completada
    • Testing realizado
    • Dependencias (si aplica)
  6. Code review por otros colaboradores (5-10 min)
  7. ✅ Approve y Merge a develop

⚠️ Órdenes de Merge (IMPORTANTE):

  • 🚨 PRIMERO: feature/admin-* → develop (cambios de BD)
  • ⏳ SEGUNDO: feature/usuario-* y feature/tecnico-* → develop (paralelo, sin orden)
  • ✅ VALIDACIÓN: Después de cada merge, ejecutar en develop:
    • php artisan migrate
    • php artisan cache:clear
    • npm run dev

📅 Ciclo por Fase:

DÍA 1 (FASE 1): 
├─ Colaborador 1: feature/usuario-limpieza
├─ Colaborador 2: feature/tecnico-limpieza  
└─ Colaborador 3: feature/admin-limpieza
   └─ Merge a develop (Admin primero)

DÍA 2-5 (FASE 2):
├─ Colaborador 1: feature/usuario-ux → develop
├─ Colaborador 2: feature/tecnico-ux → develop
└─ Colaborador 3: feature/admin-ux → develop

... continuar por cada fase ...

🚀 Deploy a Hostinger (desde main):

  1. Todos los PRs merged a develop
  2. Hacer release: git tag -a v1.1.0 -m "Release con todas las fases"
  3. git push origin v1.1.0
  4. Merge develop → main: git checkout main && git merge develop
  5. git push origin main
  6. SSH a Hostinger y hacer pull: git pull origin main
  7. Ejecutar migraciones y compilar assets

🔐 Reglas GitHub (OBLIGATORIO):

  • ✅ Protección de main: No pushear directo, solo PRs
  • ✅ Protección de develop: Requerir 1 approval antes de merge
  • ✅ Status checks: Tests pasen antes de merge
  • ✅ Auto-delete branches: Eliminar rama después de merge
